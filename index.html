03Reservation savings on Azure Plan

Dev/Test can't use reserved instances — Azure Plan (CSP) can. Committing steady-state compute to a 1- or 3-year reservation cuts the CSP rate well below PAYG, which is often what tips the move. 06What the numbers can't show

  • No SLA on Dev/Test. Dev/Test offers carry no financially-backed SLA. Azure Plan (CSP) restores standard SLAs — worth real money if these workloads matter.
  • Production is prohibited on Dev/Test. If any of this estate is quietly running production, moving to CSP is a compliance fix, not just a commercial choice.
  • Visual Studio subscription dependency. Dev/Test rates require active Visual Studio subscribers. CSP removes that dependency and its per-seat cost.
  • Marketplace & purchase charges are excluded from the repricing — they bill identically on either offer.
  • SQL Server on Dev/Test also bills without the SQL licence meter. If the estate includes SQL VMs, the true CSP cost is higher than shown unless AHB covers SQL too.
  • Reservations are modelled from public retail rates (or, where the Retail Prices API is unreachable, estimated from the export's PAYG rate and the assumed discounts you set) and assume steady-state utilisation of the reserved instances. Real commitments carry cancellation/exchange rules and capacity considerations, and a partner may pass additional margin on top — treat the reservation saving as indicative, and size it to genuinely predictable base load only. Savings plans (not modelled here) are an alternative with more flexibility and typically slightly smaller discounts.
